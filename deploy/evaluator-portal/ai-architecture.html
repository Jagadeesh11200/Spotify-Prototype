AI-Native Architecture

AI turns discovery from a replay predictor into an intent-understanding system.

The Android prototype demonstrates the front-end behavior of an AI-native discovery layer. The production system would combine review analysis, user research signals, mood intent, and listening context to recommend music that breaks repetitive habits without making the user work harder.

Why AI

Traditional recommendations optimize similarity; this problem needs intent interpretation.

Collaborative filtering and repeat-play signals are strong at predicting what a listener already likes. That strength can also trap users in loops: familiar playlists, known artists, and safe tracks. AI is useful here because it can interpret messy human language, infer emotional goals, cluster unmet needs, and translate a vague state such as "I want something warm but new" into a discovery path.

Review Discovery Engine

Part 1 backend workflow

1. Gather Signals

Collect App Store reviews, Play Store reviews, Reddit threads, community posts, forums, and social conversations.

2. Normalize And Enrich

Clean text, remove duplicates, detect platform and segment hints, and attach metadata such as region, date, sentiment, and product surface.

3. Cluster Pain Points

Use embeddings and LLM labeling to group themes like stale recommendations, algorithm fatigue, playlist dependency, and unclear discovery intent.

4. Retrieve Evidence

Store themes and source snippets in a searchable RAG index so the PM can ask why users repeat music and inspect supporting evidence.

Across MVP Phases

How AI shows up in the product experience

Phase 1: Baseline Context

  • Models the existing Spotify loop: history, likes, search, library, player, and repeated listening.
  • Creates the behavioral context the AI layer needs before suggesting a discovery intervention.
  • Shows why pure history can overfit to familiar tracks.

Phase 2: Mood Intent

  • Converts a lightweight mood choice into a recommendation brief.
  • Uses emotion, energy, valence, genre, and memory cues as richer signals than clicks alone.
  • Creates the "Right Now For You" discovery surface from current intent, not only past behavior.

Phase 3: Adaptation

  • Tracks repeated exits so the intervention becomes compact instead of intrusive.
  • Preserves normal listening while keeping discovery one tap away.
  • Turns user response into product calibration: when to ask, when to step back, and when to resurface.
Production Backend

Recommended service architecture

  1. Ingestion jobs collect public review/community data and anonymized in-product feedback.
  2. An LLM pipeline classifies discovery complaints, unmet needs, desired listening outcomes, and user segments.
  3. A vector store supports RAG queries for the PM workflow and theme evidence in the final deck.
  4. A mood-intent service converts user-selected states into structured recommendation constraints.
  5. A ranking layer blends existing recommendation candidates with AI-derived intent constraints and novelty controls.
  6. A feedback loop watches skips, plays, saves, exits, and repeat returns to tune when the Mood Board appears.
What The Prototype Demonstrates

The MVP proves the experience layer before full-scale model integration.

The current Android build uses local mock data and deterministic scoring to demonstrate the user journey: mood capture, calibrated recommendation surfaces, player continuity, library continuity, and adaptive entry behavior. The fellowship evaluation should read this as a functional MVP of the AI-native interaction model, with the review-analysis workflow and deck explaining the broader data system behind it.
